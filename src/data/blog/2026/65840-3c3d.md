## 1 实验目标

实现 Raft 的持久化（persistence）和日志压缩（log compaction）功能，确保服务器在崩溃重启后能够恢复状态并继续工作。

## 2 核心实现细节

### 2.1 需要持久化的状态字段

> If a Raft-based server reboots it should resume service where it left off. This requires that Raft keep persistent state that survives a reboot. The paper's Figure 2 mentions which state should be persistent.
>
> 如果基于 Raft 的服务器发生重启，它应能够从中断处恢复服务。这就要求 Raft 必须维护能够跨越重启的持久化状态。论文中的图 2 列出了需要持久化的状态。

当基于 Raft 的服务器崩溃或重启后，必须能够从之前的状态恢复，继续提供服务。这意味着某些关键状态必须被持久化到磁盘（survive a reboot）。如下图所示，根据 Raft 论文 Figure 2，必须持久化的状态包括：

- `currentTerm` - 服务器当前任期号，同时也是服务器所见过的最新任期（首次启动时初始化为 0，随后单调递增），重启后必须恢复。
- `votedFor` - 当前任期中获得投票的候选者 ID（若无则为 null），防止重复投票
- `log[]` - 日志条目，包含所有已接收的日志。每个条目包含状态机的命令以及该条目被领导者接收时的任期（起始索引为 1）。

![Raft 论文 Figure 2](@/assets/images/posts/2026/65840-3c3d/persistent_state.png)

实现注意点：

1. 何时持久化 - 这些状态改变时必须立即写入磁盘（即调用 persist() ）
1. 何时恢复 - 服务器启动时从磁盘读取并恢复这些状态
1. 不要持久化 - commitIndex、lastApplied、nextIndex[]、matchIndex[] 是易失性状态，重启后可重新计算
1. 易失状态推荐最后改。即使 persist() 前崩溃，磁盘状态保持一致（都是旧值），重启后逻辑正确。如：

```golang
rf.currentTerm = reply.Term
rf.votedFor = -1
rf.persist()              // ← 持久化状态一起落盘
rf.nodeState = Follower   // ← 易失状态最后改
```

特别需要注意的是，`lastIncludedIndex` 和 `lastIncludedTerm` 也需要持久化。它们是**快照的逻辑边界锚点**，既用于在 `InstallSnapshot` 和 `AppendEntries` RPC 中执行一致性检查（判断快照新旧、验证日志连续性），又决定了节点重启后状态机的恢复位置及 `commitIndex`/`lastApplied` 的初始值；若丢失这两个值，节点将无法安全地截断旧日志、正确应用新日志或与集群保持一致，从而破坏 Raft 的安全性保证。论文原文如下：

> Raft also includes a small amount of metadata in the snapshot: the last included index is the index of the last entry in the log that the snapshot replaces (the last entry the state machine had applied), and the last included term is the term of this entry. These are preserved to support the AppendEntries consistency check for the first log entry following the snapshot, since that entry needs a previous log index and term.

![](@/assets/images/posts/2026/65840-3c3d/persistent_state_snapshot.png)

参考代码如下：

```go
type Raft struct {
    // ... 其他字段

    // 持久化状态 (所有服务器)
    currentTerm int        // 最新任期
    votedFor    int        // 当前任期投票给的候选人 (-1 表示未投票)
    log         []LogEntry // 日志条目

    // 快照状态 (需要持久化)
    lastIncludedIndex int // 快照中包含的最后一个日志条目的索引
    lastIncludedTerm  int // 快照中包含的最后一个日志条目的任期

    // 易失性状态 (不需要持久化)
    // commitIndex, lastApplied, nextIndex, matchIndex...
}
```

### 2.2 nextIndex回退优化

> If desired, the protocol can be optimized to reduce the number of rejected AppendEntries RPCs. For example, when rejecting an AppendEntries request, the follower can include the term of the conflicting entry and the first index it stores for that term. With this information, the leader can decrement nextIndex to bypass all of the conflicting entries in that term; one AppendEntries RPC will be required for each term with conflicting entries, rather than one RPC per entry. In practice, we doubt this optimization is necessary, since failures happen infrequently and it is unlikely that there will be many inconsistent entries.
>
> 如果需要，可以对该协议进行优化，以减少被拒绝的 AppendEntries RPC 数量。例如，当追随者（follower）拒绝一个 AppendEntries 请求时，它可以返回冲突日志条目所在的任期（term）以及该任期中它所存储的第一个日志条目的索引。利用这些信息，领导者（leader）可以将 nextIndex 递减，从而跳过该任期内所有冲突的日志条目；这样，每个存在冲突日志条目的任期只需要一次 AppendEntries RPC，而不是每个冲突条目都需要一次 RPC。在实际应用中，我们怀疑这种优化并非必要，因为故障发生的频率很低，而且不太可能出现大量不一致的日志条目。

当 Leader 的日志与 Follower 不一致时，Leader 每次只递减 nextIndex 一个位置，然后重试。如果有多个不一致的条目，就需要多次 RPC。

优化方案：
Follower 拒绝时返回两个信息：

1.  冲突条目中的任期号（如果有）2. 该 term 的第一个索引 3. 日志长度

Leader的逻辑：1. 如果Leader没有对应的冲突任期号，nextIndex = 该 term 的第一个索引2. 如果Leader有对应的冲突任期号，nextIndex = 冲突任期号的第一个索引3. 如果Follower的日志长度小于nextIndex，nextIndex = 日志长度

这样 Leader 可以直接将 nextIndex 跳到该 term 的起始位置，一次跳过所有相同 term 的冲突条目。

参考代码：

```go
// 3C优化，加快nextIndex更新
if reply.XTerm >= 0 {
    // 如果 leader 存在 XTerm 对应的日志条目，则直接跳到该任期的末尾
    found := false
    for i := rf.lastLogIndex(); i >= rf.lastIncludedIndex; i-- {
        if rf.logTerm(i) == reply.XTerm {
            // Case 2: Leader 有 XTerm，跳到该 term 最后一个条目之后
            rf.nextIndex[server] = i + 1
            found = true
            break
        }
    }
    if !found {
        // Case 1: Leader 没有 XTerm，跳到 XIndex
        rf.nextIndex[server] = reply.XIndex
    }
} else {
    // Case 3: Follower 日志太短
    rf.nextIndex[server] = reply.XLen // XLen
}
```
